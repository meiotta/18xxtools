## Changes I just made (2026-04-09)

### js/tile-geometry.js — parseDSL() added + HEX_INRADIUS fix

**HEX_INRADIUS corrected from 43.3 → 43.5.**
Source: hex.rb Y_B=87 at scale 100 → 43.5 at scale 50. This was slightly wrong before.
Edge midpoints now use correct values (LL x = -37.67, not -37.50).

**parseDSL(dslString, color) added to TILE_GEO.**
Converts tobymao/18xx-master tile.rb DSL strings directly into the new {nodes, paths} format
for normalizeTileDef(). Full spec handled:

- city, town, offboard, junction nodes with revenue, slots, style, groups, hide, loc
- path with a, b, terminal (1 or 2), ignore, lanes, a_lane, b_lane, track
- label component
- Phase-variable revenue: `yellow_10|green_20|brown_30` → takes first value
- loc: center | N (edge midpoint) | N.5 (corner vertex between edges N and N+1)
- upgrade, border, icon, frame: silently ignored (not needed for rendering)

**Edge numbering:** parseDSL uses tobymao numbering (0=Bottom, clockwise) directly —
same as our tile defs in constants.js and tile-geometry.js. No conversion needed here.
The (tobymaoEdge+1)%6 conversion only applies in import-ruby.js for .rb map files.

**lanes:N handling:** The DSL field is stored on the path object. Expansion into N parallel
paths is deferred to render time (or a future expand step) — don't expand at parse time
since that complicates the data model.

Usage:
```js
const tileDef = TILE_GEO.normalizeTileDef(
  TILE_GEO.parseDSL(
    'town=revenue:10;path=a:0,b:_0;path=a:3,b:_0',
    'yellow'
  )
);
```

This enables tile.rb to be the direct import source for Phase 2 bulk tile conversion.

## Changes I just made (2026-04-10, session 2)

### White-tile feature system — bug fixes

The drag-drop system for white map features (city circles, OO, town dots) had several bugs that are now fixed:

1. **`city-1` tool never cleared** — `applyTool()` had a `city-1` branch that placed the feature but forgot to call `_clearWhiteTileTool()`. The tool stayed active permanently, so every subsequent hex click "activated" the panel but also appeared to paint an invisible city-1 (because the hex was being overwritten on each click). Fixed: `_clearWhiteTileTool()` now called in the `city-1` branch.

2. **Drop handler was restoring the stale tool** — After a drag-drop, the handler did `const prevTool = activeTool; activeTool = payload; applyTool(id); activeTool = prevTool;`. The `activeTool = prevTool` line undid the `_clearWhiteTileTool()` that `applyTool` just did, leaving `activeTool` as whatever it was before the drop. Fixed: removed the prevTool save/restore.

3. **City-3 rendered identically to City-2** — The renderer used `if (slots >= 2)` which always drew 2 circles for both 2-slot and 3-slot cities. Fixed: added `if (slots >= 3)` branch that draws 3 circles in a triangle formation (top-center, bottom-left, bottom-right) inside a taller rounded-rect frame.

4. **City name section invisible for white-tile features** — `updateHexPanel()` only showed the city name input when a numbered tile (`hex.tile`) was placed with `TILE_DEFS[tile].city` or `.oo`. White-tile features (`hex.city`, `hex.oo`) never triggered it. Fixed: condition is now `(td && (td.city || td.oo)) || !!hex.city || !!hex.oo`.

5. **City name not drawn on canvas for white-tile features** — Same logic gap in renderer.js. Fixed with `const hasCityOrOo = (tileDef && ...) || !!hex?.city || !!hex?.oo`.

6. **Right-panel scrollbar eating into content width** — `overflow-y: scroll; scrollbar-gutter: stable` was reserving scrollbar space but `box-sizing: border-box` meant the content area shrank. Replaced with `overflow-y: auto` plus `::-webkit-scrollbar` rules for a 6px overlay-style scrollbar that doesn't steal content space.
